# old.diemer.codes

The original personal site — a Create React App from 2019 (React 16.12,
`react-scripts` 3.2.0, styled-components 4.4.1) — brought back up on the cluster
at <https://old.diemer.codes>, kept exactly as it was.

```
browser ──TLS──▶ Cloudflare edge ──tunnel──▶ infra/cloudflared
                                                   │ https, No TLS Verify
                                                   ▼
                              traefik.kube-system.svc.cluster.local:443
                                                   │ routes by Host header
                                                   ▼
                                     nginx (this chart, site baked in)
```

Also reachable at <https://old.zachd.duckdns.org>, which rides the existing
DuckDNS wildcard cert and does not depend on Cloudflare.

## Why this chart is shaped backwards

The repo's convention (root README, "Conventions") is that apps we write live in
their own repo owning **both** their chart and their source, submodule'd back
here. This one inverts that: the chart and the Dockerfile live *here*, and the
app is the submodule underneath at `site/`.

That's deliberate. [zdiemer/diemer.codes](https://github.com/zdiemer/diemer.codes)
is a frozen archive and the whole point of this exercise was to redeploy it
**without modifying it**. So everything it would normally own — Dockerfile,
nginx.conf, chart — lives in this directory instead.

Two consequences worth knowing:

- **The submodule has to sit inside the chart directory.** `docker build` cannot
  `COPY` from outside its context, and the context is this directory. Anywhere
  tidier either breaks the build or forces the context up to the repo root,
  dragging all of `selfhosted` to the daemon.
- **The constraint holds mechanically, not by discipline.** npm only ever runs
  inside the Docker build, so `site/` is never dirtied by building. `git status`
  will not sprout `modified: site (untracked content)`.

Note the "source lives in the chart dir" part isn't a deviation at all —
`games/gamedex` already has `src/`, `static/`, `data/` and its Dockerfile beside
its chart. The only inversion is *which repo owns the chart*.

## The landmine: `npm run build` ships a silently unstyled site

The single most important thing in this directory. `public/index.html` links
`%PUBLIC_URL%/site.css`. That file:

- is generated by `lessc` via the `build-css` script,
- is gitignored upstream (`public/site.css*`), so it is not in the repo,
- and **is not produced by `npm run build`** — only the (broken) `start` script
  ever chained `build-css` into the build.

CRA copies `public/*` into `build/` verbatim, so `site.css` must exist *before*
`build` runs. Get the order wrong and **nothing complains**: react-scripts
succeeds, the image builds, nginx returns `200`, probes pass, no log line
anywhere — and the page renders with zero styling.

Because no normal signal catches it, there are three gates:

1. the assertion block at the end of the Dockerfile's build stage,
2. a re-check in `build.sh` against the final image, before it pushes,
3. `curl .../site.css | grep about-me-body` in the verification steps below.

If the site ever comes up unstyled, it is this. Nothing else.

## Two traps

- **Never add `NODE_OPTIONS=--openssl-legacy-provider`.** It looks like the
  obvious safety net for the Node/OpenSSL issue below. Node registers that flag
  only when built against OpenSSL 3, so on `node:16-bullseye` node refuses to
  start with `bad option`. It is the fix for building this on Node 18/20, and
  only there.
- **`less` is pinned to 4.2.0, and must not go back to 3.x.** `site.less:236`
  has `@media screen and (max-device-aspect-ratio: 10/16)` — the only bare
  division in the file. Less 3 defaulted to math-always and rewrites `10/16` to
  `0.625`, silently breaking that media query. Less 4 defaults to
  parens-division and leaves it alone.

## Other things that are already true, and not your bug

- **Node 16 is required.** `react-scripts` 3.2 → webpack 4 → md4 hashing, which
  OpenSSL 3 (Node 17+) removed. Newer Node dies with
  `error:0308010C:digital envelope routines::unsupported`.
- **`src/Site.jsx:20`'s `import "./resources/styles/site.less"` is a no-op.** CRA
  3.2 has no less-loader, so it falls through to file-loader and returns an
  unused URL string. Do not "fix" it by adding less-loader — the real CSS path is
  the `lessc` script.
- **`server.js`, `express` and `npm start` are dead.** `server.js` was deleted in
  `bef716b`; the very next commit pointed `start` at it anyway. Ignore all three.
- **The phone number in `src/Site.jsx:61` is published on purpose.** It was
  public on this site from 2018–2019; the decision was to restore the page as it
  was.

## What the image changes, and why

`overlay/index.html` replaces `site/public/index.html` at build time. It is the
only content deviation from upstream, and it exists because the fixes could not
go in the site repo:

- **Google Analytics removed.** The gtag block for `UA-118747915-1` — Universal
  Analytics stopped processing data in July 2023, so it collected nothing and was
  purely a third-party request plus a cookie on every load.
- **Fira Code actually loads now.** `site.less` asks for it in three places but
  only ever imported it from `cdn.rawgit.com`, which shut down in 2019 — and that
  `@import` sits after all other rules, which is invalid CSS that browsers ignore
  regardless. The font had been silently falling back to `sans-serif` for years.
  The overlay adds a Google Fonts `<link>`; the existing `font-family` rules pick
  it up. Weights 300/400/700, matching the old rawgit bundle (`site.less` asks
  for `font-weight: 100`, which has no Fira Code equivalent and snaps to 300).

## Usage

First time:

```bash
git submodule update --init web/old-diemer-codes/site
docker login ghcr.io -u zdiemer          # PAT with write:packages
./build.sh
# First push only: set the GHCR package visibility to Public, or all 10 nodes
# get ImagePullBackOff with a 401.
./upgrade.sh
```

Shipping a change:

```bash
# 1. commit it on `develop` in the site repo, then move the pin:
git submodule update --remote web/old-diemer-codes/site
# 2. bump the tag — values.yaml image.tag AND Chart.yaml version/appVersion.
#    imagePullPolicy is IfNotPresent, so reusing a tag will NOT re-pull.
./build.sh && ./upgrade.sh
# 3. record what shipped:
git add .gitmodules web/old-diemer-codes/site && git commit
```

## Verifying

```bash
helm lint . && helm package . -d /tmp && ls -lh /tmp/old-diemer-codes-1.0.0.tgz
```

Expect **single-digit KB**. Megabytes means `site/` is leaking past
`.helmignore`, and `helm upgrade` will fail with `Secret ... data: Too long`
(Helm packs the chart dir into the release Secret; etcd caps it at 1 MB).

```bash
curl -s  https://old.diemer.codes/site.css | grep -q about-me-body && echo STYLED
curl -sI https://old.diemer.codes/ | grep -i cache-control        # no-store
curl -sI https://old.diemer.codes/static/js/*.chunk.js | grep -i cache-control  # immutable
```

Then look at it in a browser — styling is the one thing no status code reports.
Dark background, photo behind, Fira Code text. White and unstyled ⇒ `site.css`.

## Notes

- **Cloudflare hostname mapping is not in this chart.** The tunnel's
  hostname → origin map lives in the Cloudflare dashboard (Zero Trust → Networks
  → Tunnels). `old.diemer.codes` points at
  `https://traefik.kube-system.svc.cluster.local:443`, **No TLS Verify on**, Host
  header blank. `ingress.cloudflareHosts` here only tells Traefik to answer for
  that Host. See `infra/cloudflared`.
- **Service-worker staleness is expected.** CRA's Workbox worker precaches the
  app, so after a redeploy a returning visitor sees the old site once and the new
  one on their next visit. The `no-store` on `service-worker.js` and `index.html`
  in `nginx.conf` is what makes the update land at all. Fine for a frozen
  archive.
- **The image is the reproducibility guarantee.** The 2019 dep tree resolves from
  npm today, but two FontAwesome packages are already deprecated and
  `node:16-bullseye` is EOL. Once built, the image lives on GHCR and never needs
  rebuilding. If it ever must be, and Node 16 is gone, the escape hatch is Node
  20 **plus** `NODE_OPTIONS=--openssl-legacy-provider` — which works there, and
  only there.
